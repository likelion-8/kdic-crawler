## 전일 진행 사항

### 1. RAG 파이프라인 성능 측정

전체 답변 지연 원인을 확인하기 위해 `rag_answer()`의 단계별 실행 시간을 측정했다.

기존 `rag_answer(query)`의 문자열 반환 방식은 유지하고, 내부에 `_rag_answer_traced(query)`를 추가하여 다음 단계의 실행 시간을 기록했다.

* 질문 분류
* 검색
* Reranking
* 출처 및 민원 데이터 구성
* 프롬프트 생성
* HyperCLOVA X 호출
* 전체 실행 시간

`testset_all.jsonl`에서 다음 유형의 실제 질문을 1개씩 선정하여 측정했다.

* 정보성 질문
* 민원성 질문
* 표 조회 질문
* 근거 부족 또는 범위 밖 질문

최초 모델 로딩이 포함된 Cold 실행과 모델 로딩 이후의 Warm 실행을 분리했으며, 실제 성능 비교에는 Warm 실행 결과를 사용했다.

#### 측정 결과

| 질문 유형 | Reranking | LLM 호출 |   전체 시간 |
| ----- | --------: | -----: | ------: |
| 정보성   |    99.47초 |  2.17초 | 101.90초 |
| 민원성   |   105.42초 |  2.18초 | 107.78초 |
| 표 조회  |   131.72초 |  0.98초 | 132.83초 |
| 근거 부족 |   160.78초 |  4.93초 | 166.39초 |

* 질문 분류와 검색은 대부분 1초 이내로 처리됨
* LLM 호출은 약 0.98~4.93초 소요됨
* 전체 실행 시간의 97% 이상을 Reranking이 차지함
* Cold 실행은 임베딩 모델 초기 로딩을 포함해 176.69초 소요됨

#### 병목 원인 분석

* Reranker는 요청마다 다시 로딩되지 않고 싱글턴으로 관리되고 있었음
* 배치 처리도 이미 적용된 상태였음
* 1차 검색 후보 청크 수는 20개였음
* 일부 청크는 최대 약 5,502토큰으로 매우 길었음
* Reranker의 `max_length`는 8,192였음
* 대형 Cross-Encoder를 CPU 환경에서 실행하고 있었음

따라서 지연 원인은 모델 재로딩이 아니라, **긴 청크 20개를 Cross-Encoder가 CPU에서 처리하는 연산 비용**으로 판단했다.

### 2. Reranker 도입 여부 검증

현재 CPU 환경에서 Reranking이 검색 품질을 실제로 개선하는지 확인하기 위해 적용 전후 성능을 비교했다.

#### 실험 설정

* Reranker: `BAAI/bge-reranker-v2-m3`
* 실행 환경: CPU
* `max_length`: 8192
* 1차 검색 후보: Top-20
* 최종 결과: Top-5
* 평가 문항:

  * 쉬운 질문 3개
  * 모호한 구어체 질문 3개
  * 총 6개
* 평가 지표:

  * `Recall@5`
  * `MRR@5`

동일한 1차 검색 결과를 대상으로 다음 두 방식을 비교했다.

* Reranker Off: 1차 검색 상위 5개를 그대로 사용
* Reranker On: 상위 20개를 재정렬한 뒤 상위 5개 사용

#### 실험 결과

| 구분             | Reranker Off | Reranker On |
| -------------- | -----------: | ----------: |
| 전체 Recall@5    |         0.83 |        0.83 |
| 전체 MRR@5       |         0.71 |        0.64 |
| 쉬운 질문 Recall@5 |         1.00 |        1.00 |
| 쉬운 질문 MRR@5    |         0.75 |        0.61 |
| 모호 질문 Recall@5 |         0.67 |        0.67 |
| 모호 질문 MRR@5    |         0.67 |        0.67 |

#### 주요 확인 사항

* Reranker가 새롭게 찾아낸 정답 페이지는 없었음
* `Recall@5`는 적용 전후 동일했음
* `MRR@5`는 0.71에서 0.64로 하락했음
* 기존 검색에서 정답이 1위였던 질문이 Reranking 후 2위로 내려간 사례가 있었음
* 긴 청크가 포함된 질문은 Reranking에만 약 210초가 소요됐음
* 후보 20개 Reranking에는 평균 약 27초가 소요됐음
* 정답이 처음부터 Top-20에 없는 질문은 Reranker로 개선할 수 없었음

#### 최종 결정

* Reranker를 기본 Off로 설정
* Dense·Hybrid 검색 결과를 그대로 사용
* 기본 RAG 파이프라인에서 Reranking 단계 제거
* 향후 다음 조건이 확보되면 재검토

  * GPU 실행 환경
  * 짧은 `max_length`
  * 적은 후보 수
  * FlashRank 등 경량 Reranker

평가 문항이 6개로 적다는 한계가 있으나, 현재 CPU 환경에서는 검색 품질 개선 효과보다 응답 시간 증가가 지나치게 크다고 판단했다.

### 3. 골든셋 정합성 검증 및 수정

#### `must_include` 문구 교정

실제 `corpus.jsonl`의 표현과 일치하지 않던 `must_include` 문구 총 13건을 수정했다.

주요 수정 유형은 다음과 같다.

* 표 헤더 표현 반영

  * `보호되지 않` → `비보호`
* 조사 및 어미 차이 수정

  * `별도 신청` → `신청 절차는 필요하지 않`
  * `보호받을 수 없` → `보호받으실 수 없`
  * `변경되지 않` → `변경되지는 않`
* 실제 페이지에 없는 표현 교체

  * `가능` → `사용할 수 있`
  * `각 저축은행별` → `법인이 다를 경우`
  * `법령정보시스템` → `해당페이지 이동하기`

정답 의미는 유지하면서 자동 검증이 가능하도록 실제 코퍼스 표현과 일치시켰다.

#### 문항 삭제

답변 근거 또는 데이터 상태에 문제가 있는 문항 총 3건을 삭제했다.

| test_id             | 삭제 이유                                    |
| ------------------- | ---------------------------------------- |
| `dp_logo_q3`        | 색상 정보가 이미지에만 존재하고 코퍼스 텍스트에는 없어 답변 근거가 없음 |
| `uc_hrpe_hist_q9`   | `corpus.jsonl`의 링크 필드가 크롤링 과정에서 손상됨      |
| `dp_prdct_srch_q10` | 관련 문항 검토 후 평가 대상에서 함께 제외                 |

전체 문항 수는 856건에서 853건으로 변경됐다.

### 5. Streamlit UI 연동 및 오류 개선

#### 발생 문제

* CLI 환경에서는 정상 동작했지만 Streamlit 연동 후 응답이 느려지거나 멈추는 현상이 발생함
* 파이프라인 반환값 형식 불일치로 다음 오류가 발생함

```text
tuple indices must be integers or slices, not str
```

#### 원인 분석

##### 멀티스레드 충돌

Streamlit의 Python 스레드와 다음 C++ 기반 스레드가 충돌했다.

* PyTorch 기반 BGE-m3
* Kiwi 형태소 분석기
* OpenMP
* MKL

이로 인해 스레드 교착 상태와 연산 지연이 발생했다.

##### 반환값 타입 불일치

`_rag_answer_traced()`는 튜플을 반환하지만, UI에서는 딕셔너리 키 방식으로 접근하고 있어 타입 오류가 발생했다.

#### 해결 조치

##### 환경변수 설정

`app.py` 최상단에 다음 설정을 추가했다.

```text
TOKENIZERS_PARALLELISM=false
OMP_NUM_THREADS=1
MKL_NUM_THREADS=1
```

C++ 기반 백그라운드 스레드 수를 제한해 충돌을 방지했다.

##### Streamlit 리소스 캐싱

`@st.cache_resource`를 적용해 다음 리소스를 앱 실행 시 한 번만 로드하도록 수정했다.

* BGE-m3 임베딩 모델
* Kiwi BM25 인덱스
* Intent 분류기

최초 1회 워밍업 이후 질문마다 모델과 인덱스를 다시 로딩하지 않도록 했다.

##### 반환값 방어적 처리

파이프라인 결과가 다음 형식 중 어떤 형태로 반환돼도 처리할 수 있도록 `isinstance()` 기반 방어 로직을 추가했다.

* `dict`
* `tuple`
* `str`

#### 최종 결과

* 최초 1회 로딩 후 CLI와 유사한 응답 속도 확보
* Streamlit 멈춤 및 지연 현상 개선
* 반환값 타입 오류 제거
* 답변, Intent, 단계별 소요시간 정상 출력 확인

발표 중 실제 서비스 흐름을 안정적으로 보여줄 수 있도록 대표 질문을 선정하고 시연 순서를 정리한다.

#### 대표 질문 유형

* 정보성 질문
* 표 조회 질문
* 민원성 질문
* 첨부파일 또는 양식 요청 질문
* 근거 부족 질문
* 범위 밖 질문

#### 시연 확인 항목

* 질문 유형과 답변 의도가 정상적으로 분류되는지
* 적절한 검색 방식이 선택되는지
* 답변 내용이 검색 근거와 일치하는지
* 출처 URL이 정상적으로 표시되는지
* 민원 질문에서 첨부파일과 공식 페이지가 표시되는지
* 근거가 없는 질문에 임의 답변을 생성하지 않는지
* 최초 모델 로딩 이후 응답이 정상적으로 처리되는지
* Streamlit 화면이 멈추거나 오류를 출력하지 않는지

실시간 시연 실패에 대비해 정상 실행 화면과 결과를 캡처해 둔다.

* `requirements.txt` 의존성 확인
* 불필요한 라이브러리 제거

* Windows 환경에서 실행 가능한지 확인
* Streamlit 실행 명령어 확인
* 초기 인덱스 및 캐시 생성 방법 확인
* 주요 스크립트 실행 순서 정리
