# RAG 파이프라인 이슈 정리

현재 파이프라인(`src/pipeline.py`)에서 확인된 두 가지 문제 — 응답 지연(Reranking)과 출처 URL 오출력 — 의 원인과 해결 방안을 정리한다.

---

## 이슈 1. Reranking 단계 CPU 병목으로 인한 응답 지연

### 목적

전체 답변 시간이 오래 걸리는 원인을 확인하기 위해 `rag_answer()`의 단계별 실행 시간을 측정한다.

### 측정 방식

- 기존 `rag_answer(query)`는 문자열을 반환하는 구조를 유지한다.
- 내부에 `_rag_answer_traced(query)` 함수를 추가한다.
- `_rag_answer_traced()`에서 단계별 시간과 최종 답변을 함께 반환한다.
- 성능 측정 스크립트에서만 `_rag_answer_traced()`를 사용한다.

측정 단계:

- 질문 분류
- 검색
- Reranking
- 출처 및 민원 데이터 구성
- 프롬프트 생성
- HyperCLOVA X 호출
- 전체 실행 시간

### 대표 질문 선정

`testset_all.jsonl`에서 실제 라벨이 확인된 질문을 유형별로 1개씩 선택한다.

- 정보성 질문
- 민원성 질문
- 표 조회 질문
- 근거 부족 또는 범위 밖 질문

새로운 질문을 임의로 만들지 않고 기존 테스트셋의 질문을 사용한다.

### Cold와 Warm

- **Cold 실행**: 프로그램 실행 후 첫 질문으로, 모델과 인덱스 로딩 시간이 포함된다.
- **Warm 실행**: 첫 실행 이후 모델과 데이터가 메모리에 올라온 상태에서 질문을 처리한다.
- 실제 질문 처리 성능은 Warm 실행 시간을 기준으로 비교한다.

첫 질문을 웜업용으로 한 번 실행한 뒤, 대표 질문 4개를 다시 실행하여 측정한다.

### 측정 결과

단계별 실행 시간을 측정한 결과, 전체 응답 시간의 대부분이 **Reranking 단계**에서 발생하였다.

| 질문 유형 | Reranking | LLM 호출 | 전체 시간 |
| --- | --- | --- | --- |
| 정보성 | 99.47초 | 2.17초 | 101.90초 |
| 민원성 | 105.42초 | 2.18초 | 107.78초 |
| 표 조회 | 131.72초 | 0.98초 | 132.83초 |
| 근거 부족 | 160.78초 | 4.93초 | 166.39초 |

- 질문 분류: 약 0.01~0.6초
- 검색: 약 0.01~0.26초
- 근거 조립: 0초에 가까움
- 프롬프트 조립: 0초에 가까움
- LLM 호출: 약 0.98~4.93초
- Reranking: 약 99~161초

전체 실행 시간의 **97% 이상을 Reranking이 차지**하고 있어, 현재 응답 지연의 원인은 거의 전부 재정렬 단계로 확인되었다.

### Cold/Warm 실행 결과

웜업 실행의 전체 시간은 176.69초였다.

이 실행에는 임베딩 모델의 최초 로딩 시간이 포함되어 있어 일반 질문 처리 시간과 직접 비교하지 않는다. 문서에는 Cold 실행 또는 웜업 결과로 별도 표시하였다.

대표 질문 4개의 성능 비교에는 모델 로딩이 끝난 이후의 Warm 실행 결과를 사용하였다.

### 원인 확인

Reranking 관련 코드 구조(`src/reranker.py`)를 확인한 결과는 다음과 같다.

- Reranker 모델은 싱글턴으로 관리되어 요청마다 다시 로딩되지 않는다.
- `device="cpu"`로 설정되어 CPU에서 실행된다.
- 질문과 후보 청크를 `model.predict(pairs)`에 전달하여 배치로 처리한다.
- 초기 후보는 20개다(`pipeline.py`의 `K_CANDIDATES=20`).
- 일부 청크는 최대 약 5,502토큰으로 매우 길다.
- Reranker의 `max_length`는 8,192로 설정되어 있다.

따라서 현재 지연은 모델 재로딩이나 반복 호출 문제가 아니라, **긴 청크 20개를 대형 Cross-Encoder가 CPU에서 처리하면서 발생하는 순수 연산 시간**으로 판단된다.

Cross-Encoder는 질문과 청크 전체를 함께 입력해 Attention을 계산하므로, 입력 길이가 길어질수록 연산량이 크게 증가한다. 특히 긴 청크가 포함된 질문에서 Reranking 시간이 100초 이상으로 증가하였다.

### 현재 결론

- 검색이나 HyperCLOVA X 호출은 주요 병목이 아니다.
- 전체 지연의 핵심 원인은 CPU 기반 Cross-Encoder Reranking이다.
- 모델 반복 로딩 문제는 확인되지 않았다.
- 배치 처리도 이미 적용되어 있다.
- 긴 청크, 높은 `max_length`, 후보 20개, CPU 실행의 조합이 주요 원인으로 보인다.

### 추가 실험 — 리랭커 도입 여부 검증 (결론: Off)

리랭킹 단계가 답변 속도를 크게 늦추는 것을 확인한 뒤(질문 1개당 약 30초, 긴 청크가 걸리면 그 이상), 속도를 튜닝하기 전에 근본 질문부터 확인하기로 했다: **리랭커가 그 시간을 쓸 만큼 검색 품질을 실제로 올리는가?**

**측정 방법**

현행 파이프라인 설정을 그대로 사용했다 — 리랭커는 `BAAI/bge-reranker-v2-m3`(cross-encoder), 1차 검색 상위 20개 후보를 재정렬해 상위 5개를 남기는 방식, `max_length=8192`, CPU 실행. 동일한 1차 후보(top-20)에 대해 "리랭크 없이 상위 5개"와 "리랭크 후 상위 5개"를 비교해, 정답 페이지가 상위 5개에 들어오는지(Recall@5)와 그 순위(MRR@5)를 페이지 단위로 측정했다. 표본은 쉬운 질문 3개 + 새로 만든 모호(구어체) 질문 3개, 총 6개.

**결과**

| 구분 | Recall@5 (Off→On) | MRR@5 (Off→On) |
| --- | --- | --- |
| 전체 6문항 | 0.83 → 0.83 | 0.71 → 0.64 |
| 쉬운 3문항 | 1.00 → 1.00 | 0.75 → 0.61 |
| 모호 3문항 | 0.67 → 0.67 | 0.67 → 0.67 |

리랭커가 새로 건져낸 정답은 하나도 없었고(Recall 개선 0), 순위는 오히려 소폭 나빠졌다.

**질문별 원인**

- 한 쉬운 질문: 리랭크 없이 정답이 이미 1위였는데, 리랭커가 2위로 강등시킴(순위점수 1.00→0.50).
- 한 모호 질문: 순위 변화 없음. 그런데도 긴 청크 때문에 리랭킹에만 210초(3.5분) 소요.
- 다른 모호 질문: 정답 페이지가 애초에 1차 검색 top-20 안에 없어 리랭커가 손을 쓸 수 없었음.
- 나머지: 변화 없음.

**해석**

임베딩(bge-m3-ko) 기반 1차 검색이 이미 충분히 강해서 리랭커가 더 얹을 여지가 거의 없고, 오히려 이미 맞게 배열된 순위를 흔드는 부작용이 관찰된다. 여기에 리랭커는 후보마다 모델을 다시 돌려야 하는 cross-encoder라 CPU에서 극단적으로 느리다(격리 측정에서 후보 20개에 약 27초, 긴 청크가 섞이면 질문 하나에 수 분까지 증가). 종합하면 현행 설정(k=20, max_length=8192, CPU)의 리랭커는 **이득은 없거나 마이너스인데 비용(시간)은 매우 크다.**

**한계**

표본이 6개로 매우 작다. 특히 MRR 하락은 사실상 한 질문(정답 강등 사례)이 주도한 것이라 통계적으로 확정할 수준은 아니다. 큰 표본으로 확인하는 게 이상적이지만, CPU·max_length=8192·k=20 설정에서는 40문항에 한 시간이 넘어 현실적으로 대규모 측정이 불가능했다. 대규모 재확인은 GPU(예: Colab)나 더 가벼운 리랭커 설정에서 별도로 진행해야 한다.

### 최종 결정: 리랭커 기본 Off

위 실험 결과를 근거로 팀 협의를 거쳐 **리랭커를 기본 Off로 결정했다.** 현 시점에서 리랭커는 검색 품질을 올리지 못하면서 응답 속도만 크게 악화시키기 때문이다.

향후 재도입을 검토한다면:

1. GPU 환경에서 더 큰 표본으로 리랭커 이득을 재측정한다.
2. `max_length`·후보 수를 줄인 경량 설정에서 품질(Recall@5·MRR@5)이 유지되는지 확인한 뒤 재도입 여부를 다시 판단한다.

### 구현 반영

위 결정을 실제 코드에 반영했다.

- **`reranker.py` → `candidate_ranking.py` 파일명 변경**
  리랭킹이 꺼진 상태에서 이 파일이 실제로 하는 일은 "1차 검색 후보를 상위 k로 좁히는 것"(`top_k_cut`)이고, cross-encoder 재정렬(`rerank`)은 있지만 안 쓰는 선택 단계다. 파일명이 여전히 `reranker.py`면 "리랭커인데 왜 안 쓰지?"라는 의문을 코드만 보고는 풀 수 없어서, 실제 역할에 맞게 이름을 바꿨다.

- **`pipeline.py`에 `USE_RERANKER = False` 플래그 추가**
  `rerank()` 함수 코드 자체는 `candidate_ranking.py`에 그대로 남겨두고, `pipeline.py`에서 이 플래그로만 호출 여부를 가른다(`rerank(query, candidates) if USE_RERANKER else candidates`). 재도입은 이 값을 `True`로 바꾸는 한 줄로 끝나며, 이 파일에 재도입 전 GPU/경량 설정 재검증이 필요하다는 근거도 주석으로 같이 남겨뒀다.

- **docstring에 결정 근거를 self-contained로 기록**
  `candidate_ranking.py` 상단에 비활성 상태·측정 결과·재도입 조건을 그대로 적어뒀다. gitignore된 별도 결정 문서(`project_context.md` 등) 없이도 이 파일만 보고 "왜 꺼져 있는지"를 알 수 있게 하기 위함이다.

- **참조 정리**: `pipeline.py`의 import(`from candidate_ranking import rerank, top_k_cut`)와 `prompt_builder.py`·`retrieval.py`의 관련 주석을 새 파일명·역할에 맞게 갱신했다.

**효과**: 질문당 27~210초 걸리던 리랭킹 단계가 사라져 응답 속도가 대폭 빨라졌고, Off 상태에서 측정했던 "무리랭크" 경로(Recall@5 0.83)와 동일한 품질을 유지한다.

**참고 — Off 결정 이전에 검토했던 속도 최적화 옵션**

리랭커를 계속 쓴다는 전제로 정리했던 대안들이다. 지금은 채택하지 않지만, 향후 재도입을 검토할 때 참고용으로 남겨둔다.

1. **`K_CANDIDATES` 축소** (`pipeline.py`, 현재 20)
   후보 수를 줄이면 Reranking 호출 횟수가 그만큼 줄어 선형적으로 빨라진다. 다만 20이라는 값은 `route_eval.py`로 실측한 Recall@20(99%+) 근거로 정해진 값이라, 줄이기 전에 더 작은 k(예: 10)에서의 Recall을 같은 방식으로 측정해 정답률 손실이 감내할 수준인지 먼저 확인해야 한다.

2. **`max_length` 축소** (`candidate_ranking.py`, 현재 8192)
   8192는 코퍼스 전체 최대 토큰(5,502)을 커버하기 위한 값이지만, 대부분의 청크는 이보다 훨씬 짧다. 청크 길이 분포를 실측해 상위 몇 %가 실제로 8192에 가까운 길이를 쓰는지 확인하고, 대다수를 커버하는 더 작은 값(예: 2048~3072)으로 낮추면 긴 꼬리 청크 일부만 잘리는 대가로 평균 연산 시간을 크게 줄일 수 있다.

3. **더 가벼운 reranker 모델로 교체**
   `bge-reranker-v2-m3` 대신 `bge-reranker-base` 등 더 작은 cross-encoder로 바꾸면 속도는 오르지만 재정렬 정확도가 떨어질 수 있다. 임베딩 모델 비교 때와 동일하게, 후보 모델들의 AnswerRecall@5를 같은 테스트셋으로 비교해 손실 폭을 확인한 뒤 결정한다.

4. **GPU 사용** (`device="cpu"` → `device="cuda"`)
   가장 근본적인 해결책이지만 코드 변경(`candidate_ranking.py`, 그리고 동일하게 CPU로 고정된 `retrieval.py`의 Dense 임베딩 모델도 함께)과 GPU 서빙 인프라가 모두 필요하다. 배포 환경에 상시 GPU를 확보할 수 있는지가 선행 조건이다.

5. **조건부 Reranking**
   1차 검색(BM25/Dense/Hybrid) 점수 격차가 이미 충분히 크면(상위 후보가 압도적으로 높은 점수) Reranking을 생략하고, 격차가 애매한 경우에만 Reranking을 적용하는 방식. 구현 난이도가 있어 우선순위는 낮지만, 평균 응답 시간을 줄이는 데는 효과적일 수 있다.

위 실험에서 리랭커가 품질 이득을 보이지 못했기 때문에, 지금은 이 옵션들을 적용하기보다 리랭커 자체를 Off 상태로 두는 것이 우선이다. 재도입을 검토하게 되면 그때 이 목록부터 순서대로(1→5) 시도한다.

## 이슈 2. 답변 출처 URL 오출력

### 문제

`rag_answer()`가 생성한 답변 끝에 붙는 출처 표시에서 URL이 누락되거나 실제 검색된 페이지와 다른 출처가 출력되는 문제가 발견되었다.

### 원인

`src/citation.py`는 문제가 없다 — `format_all_citations()`가 `data/corpus.jsonl`에서 실제 `source_url`·`page_title`을 정확히 조회해 `prompt_builder.py`에 넘긴다.

문제는 `src/prompt_builder.py`의 `FEW_SHOT_EXAMPLES`에 있다:

```python
"answer": "주민등록증·운전면허증·여권 등 공공기관 발행 신분증과 본인의 도장(서명 가능)만 있으면 됩니다.\n\n(출처: 신청시 구비서류 안내)",
```

세 개 few-shot 예시 전부 답변 끝을 `(출처: <제목>)` 형식으로 마무리하는데, **URL이 전혀 포함되어 있지 않다.** 반면 `SYSTEM_INSTRUCTION`에는 "답변 끝에 참고한 출처(페이지 제목과 링크)를 안내하세요"라고 되어 있어, **지시문과 few-shot 예시의 요구 포맷이 서로 다르다.**

실제 프롬프트 구조를 보면, 진짜 검색 결과의 제목·URL은 `참고 출처:` 섹션에 별도로 주어지는데(`build_informational_prompt()`의 `sources` 변수), 모델이 이 섹션을 참고해 답변을 구성하기보다 **few-shot 예시의 답변 종료 패턴(제목만 있는 짧은 문구)을 그대로 흉내 내면서** URL을 빠뜨리거나, 심한 경우 예시에 나온 고정 문구("신청시 구비서류 안내")를 실제 질문과 무관하게 재사용하는 것으로 보인다. LLM은 일반적으로 추상적인 지시문보다 few-shot 예시의 출력 형식을 더 강하게 따라가는 경향이 있어, 이런 불일치가 있으면 예시 쪽이 우선하는 경우가 많다.

### 해결 방안

1. **`FEW_SHOT_EXAMPLES`의 답변 형식을 실제 원하는 출력 형식으로 수정** (가장 중요)
   세 예시 모두 답변 종료 문구를 `(출처: 제목)`이 아니라 `(출처: 제목 - URL)` 또는 팀이 최종적으로 원하는 형식(예: 마크다운 링크)으로 바꿔서, 모델이 URL 포함 형식을 그대로 학습하도록 한다. 이때 예시에 쓰는 URL도 실제 `data/corpus.jsonl`에 있는 값을 그대로 가져와야 한다(지어내면 안 됨 — 기존에도 `reference_answer`를 그대로 가져다 쓴 원칙과 동일).

2. **`SYSTEM_INSTRUCTION`과 few-shot 예시 포맷 일치**
   지시문 3번("페이지 제목과 링크를 안내하세요")과 예시가 항상 같은 형식을 보여주도록 맞춘다. 포맷이 하나로 통일되면 모델이 헷갈릴 여지가 줄어든다.

3. **"참고 출처" 섹션을 그대로 인용하라는 지시 강화**
   `build_civil_petition_prompt()`에는 이미 "위 URL은 새로 만들지 말고 그대로 인용하세요"라는 문구가 있다. `build_informational_prompt()`에도 동일한 취지의 문구를 추가해, 모델이 `참고 출처:` 섹션의 실제 URL을 답변에 그대로 반영하도록 명시적으로 강조한다.

**1번(few-shot 예시 수정)만으로도 대부분 해결될 가능성이 높고, 2·3번은 재발 방지를 위한 보강책이다.**

## 이슈 2 후속 — 실제 적용한 해결책 (2026-07-23)

위에서 제안한 방법(few-shot 형식 수정 등)을 실제로 시도해봤지만, 재현 테스트에서 충분히 해결되지 않았다. 그래서 방향을 바꿔 더 구조적인 방식으로 해결했다.

### 시도했지만 부족했던 것들

다음 세 가지를 순서대로 적용해봤다.

1. `SYSTEM_INSTRUCTION`에 "few-shot은 형식만 참고하고, 근거에 없는 내용은 가져오지 말 것"이라는 원칙 추가
2. 절차 → 서류 → 페이지 순서 규칙에 "근거 없는 단계는 생략" 예외 추가
3. 서류·링크 근거가 비어있으면 그 섹션 자체를 프롬프트에서 삭제(빈 칸 자체를 안 줌)

문제였던 실제 질문("나의 미수령금", "나 돈 잘못보냈는디 어캄쇼" 같이 짧고 모호한 질문)으로 다시 테스트해보니, 세 가지를 다 적용해도 여전히:

- 근거에 없는 서류명이나 전화번호를 지어내고
- 심지어 우리가 준 적도 없는 URL(`https://www.kdic.or.kr/vaup/...`)을 새로 만들어냈다.

질문이 짧고 few-shot 예시와 주제가 비슷할수록 이 현상이 심했다. 프롬프트 지시문을 아무리 강하게 써도 LLM이 지시문보다 few-shot의 "내용"을 더 강하게 따라가는 경향이 남아있어서, **프롬프트 문구만으로는 근본적으로 막기 어렵다**고 결론 내렸다.

### 실제 해결책: URL을 LLM에게 아예 안 보여준다

"LLM이 URL을 안 틀리고 잘 쓰게 만들기"가 아니라, **"LLM이 URL을 아예 쓸 일이 없게 만들기"**로 접근했다.

- `build_informational_prompt()` / `build_civil_petition_prompt()`: 프롬프트에서 서류·링크·출처 URL을 완전히 뺐다. LLM은 이제 "절차 설명"이나 "근거 기반 답변 본문" 텍스트만 쓴다.
- `SYSTEM_INSTRUCTION`에 "URL·웹사이트 주소·전화번호를 답변에 직접 쓰지 마세요. 그 부분은 시스템이 별도로 붙여줍니다"라는 원칙을 추가했다.
- `assemble_informational_answer()` / `assemble_civil_petition_answer()` (신규 함수): LLM이 쓴 답변 텍스트 뒤에, `citation.py`/`civil_petition.py`가 검색 결과에서 이미 조회해둔 **진짜** documents/links/citations를 그대로 붙인다. 이 부분은 LLM이 전혀 관여하지 않는 100% 결정론적 코드라서, 애초에 지어낼 수 있는 여지가 없다.
- `pipeline.py`: LLM 호출 직후 위 assemble 함수로 최종 답변을 조립하도록 흐름을 바꿨다.

### 곁들여 고친 것 — 다운로드 URL 자체가 깨져있던 문제

서류 URL을 들여다보다가 별개의 문제도 발견했다. `civil_petition.py`가 넘겨주던 다운로드 링크(`resolved_url`)가 애초에 작동하지 않는 링크였다. 코퍼스 전체 103개 서식 다운로드 항목을 확인해보니 `resolved_url`이 딱 3개의 URL로만 겹쳤다 — 이건 실제 파일 링크가 아니라 자바스크립트 다운로드 버튼이 공통으로 쓰는 POST 전용 서블릿 주소였다. 실제 파일 구분은 암호화된 파라미터를 POST 본문으로 같이 보내야만 되는데 우리는 그 값을 안 보내고 있어서, 클릭해도 다운로드가 안 되는 깨진 링크를 안내하고 있었던 것이다. `resolved_url` 대신 다운로드 버튼이 실제로 있는 페이지 주소(`page_url`)를 안내하도록 수정했다.

### 검증 결과

문제였던 질문 2개와 정상 케이스 2개, 총 4개 질문으로 재확인했다. 4개 전부 URL 관련 할루시네이션이 완전히 사라졌고, 백엔드가 붙인 링크는 항상 실제 검색 결과와 정확히 일치했다.

| 질문 | 결과 |
| --- | --- |
| 나의 미수령금 | URL을 지어내던 문제 해결 - 신청 페이지 5건이 실제 링크로 정확히 표시됨 |
| 나 돈 잘못보냈는디 어캄쇼 | 깨끗한 절차 설명 + 정확한 링크 5건 |
| 위임장 양식 (회귀 확인) | 서류 12건 + 링크 2건 정확히 붙음, 이상 없음 |
| 예금보호한도 (회귀 확인) | 출처 2건 정확히 붙음, 이상 없음 |

### 남은 사소한 여지

LLM이 절차 설명 안에서 가끔 "신분증과 도장" 같은 일반적인 표현을 근거에 정확히 없어도 관습적으로 언급하는 경우가 있었다(예: 미수령금 답변). 다만 이건 URL이나 구체적인 서류명을 지어내는 것보다는 훨씬 가벼운 문제라, 지금은 별도 이슈로 남겨두고 범위 밖으로 둔다.
