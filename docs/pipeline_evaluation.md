# 파이프라인 자동 평가 (evaluate_pipeline.py)

`data/testset/testset_all.jsonl`(856문항)을 RAG 파이프라인에 전부 흘려 답변을 받고, 정답
데이터와 대조해 품질 지표를 낸다. 팀원이 그대로 재실행·재현할 수 있도록 커맨드·지표 정의·
검증 과정·한계를 여기 정리한다.

> ⚠️ 이 문서의 수치는 2026-07-23 실행 결과다. 코퍼스·테스트셋·프롬프트가 바뀌면 달라지니,
> 인용 전 아래 커맨드로 재실행해 확인할 것. 채점 **로직**은 `--selftest`로 언제든 검증 가능.

## 1. 실행 커맨드

```bash
# 채점 로직만 검증 (모델·HCX 불필요, 항상 먼저 이걸로 코드 확인)
python3 src/evaluate_pipeline.py --selftest

# 소규모 확인 (앞 N문항) — 전량 전에 반드시 먼저
python3 src/evaluate_pipeline.py --limit 50

# 전량 실행 (문항당 HCX 1회)
python3 src/evaluate_pipeline.py

# 답변 정확도 LLM judge 포함 (in-scope 문항당 HCX 1회 추가)
python3 src/evaluate_pipeline.py --judge

# judge 신뢰도 교차검증 (이미 만든 결과를 읽음 — 재실행/HCX 불필요)
python3 src/evaluate_pipeline.py --validate-judge
```

### 재현 시 주의 (안 지키면 실패/오염)
- **Qdrant 임베디드는 단일 프로세스.** 평가 도는 동안 챗봇(`python3 src/pipeline.py`) 터미널을
  열지 말 것 — 락 충돌로 죽는다.
- **HCX 키는 repo 루트 `.env`** 에 있어야 한다(`llm_client.py`가 거기서 읽음). 커밋 금지.
- **HCX RateLimit(429).** 856 전량 시 일부 문항이 429로 실패할 수 있다(코드 오류 아님).
  실패 문항은 `failed_cases.jsonl`에 격리되니 그 test_id만 재실행해 채우면 된다.
- 첫 실행은 임베딩 모델(`dragonkue/BGE-m3-ko`) 로딩/다운로드가 있다.

## 3. 지표 정의와 변별력

채점 함수는 전부 순수 함수라 `--selftest`로 검증된다. **지표마다 변별력이 다르니 아래 구분을
알고 읽을 것.**

| 지표 | 정의 | 성격 |
|---|---|---|
| 답변 생성 성공률 | 오류 없이 답변이 나온 비율 | 실측 |
| 평균 응답시간 | timings.total 평균(judge 제외) | 실측 |
| intent 라우팅 정확도 | 예측 intent == 라벨 | **변별력 ○** |
| 정답 출처 포함률 | 사용한 top 청크 페이지에 정답 페이지가 있나(복수정답 허용) | **변별력 ○ (핵심)** |
| 출처 포함률 | 답변에 출처 URL(http)이 붙었나 | 메커니즘 확인용(항상 ~1.0) |
| **답변 정확도 (judge)** | 기준답변 대비 사실 부합(HCX judge) | **변별력 ○, 단 아래 4장 참고** |
| must_include 커버리지 | 필수문구가 **전부** 답변에 있나(공백·날짜 정규화) | **하한선** — 4장 참고 |
| 거절 성공률(oos) | out_of_scope 문항을 거절했나 | **변별력 ○** |
| 과잉 거절률(in-scope) | 답할 수 있는데 거절했나 | **변별력 ○** |
| 민원 절차/서류/공식페이지 포함률 | 각 섹션이 답변에 있나(섹션 heading 판정) | 절차는 항상 ~1.0, 서류/페이지는 변별력 ○ |

## 4. 답변 정확도: 프록시와 judge, 그리고 검증

"답변 정확도"는 두 가지로 잰다. **둘 다 완벽하지 않으니 교차로 봐야 한다.**

- **must_include 커버리지 (결정론 프록시)** — 사람이 정한 필수문구가 답변에 다 있나를 정확매칭.
  공백·날짜 표기차는 `_normalize`로 흡수한다(`"익 영업일"`≡`"익영업일"`, `"2012년 9월 10일"`≡
  `"2012-09-10"`). **all-or-nothing이라 하한선**이다 — 필수문구 2개 중 1개를 의미상 맞게
  풀어써도 0점이 된다. 그래서 실제 정확도보다 **낮게** 나온다.
- **LLM judge (--judge)** — 기준답변과 후보답변을 HCX에 같이 넣어 "사실적으로 부합하나"를 예/아니오로
  받는다. 표현차를 흡수해 프록시보다 실제에 가깝지만, **판정 주체가 LLM이라 그 자체를 검증해야 한다.**

### judge 신뢰도 검증 방법 (`--validate-judge`, 재현 가능)

judge를 결정론 프록시와 교차대조하고, **판단이 갈리는 지점**을 층화 표본으로 뽑아 사람이 직접
확인한다. 커맨드 한 번이면 교차표(`judge_validation.json`)와 표본(`judge_review_sample.jsonl`)이
나온다. 표본의 `사람판정` 칸을 검토자가 채워 judge가 맞았는지 기록하면 된다.

**2026-07-23 실행 결과 (재현 시 값은 재확인):**
- judge 정확도 0.75 vs 프록시(정규화 후) 0.48, 교차 일치율 0.67.
- 최대 불일치 버킷 `judge=정답/프록시=누락` 241건 — 표본 확인 결과 **대부분 프록시의 형식·부분답변
  과소평가**였다(정규화로 형식 오탐 45건은 이미 해소). 즉 **프록시가 정확도를 과소평가**하고,
  judge(0.75)가 실제에 더 가깝다.
- 다만 표본 채점에서 judge는 **약한 관대편향**을 보였다 — 필수 세부 하나를 빠뜨린 답변을 통과시키는
  경우가 있었다. 따라서 judge 0.75는 "완전정답" 기준으론 소폭 과대추정. **소수점 2자리로 단정하지
  말고 "대략 75%±"로 읽고**, 표본 검토로 보정할 것.

## 5. 이 평가로 드러난 품질 이슈 (지표 신뢰도와 별개, P2 개선 과제)

- **거절이 잘 안 됨** — out_of_scope 문항의 상당수를 거절하지 않고 답한다. 근거 없는 답변 =
  민원 리스크(설계 3제약 中 2)와 직결. oos 표본 수가 작으니(31건) 먼저 oos 문항을 늘려 재측정 권장.
- **과잉 거절** — 답할 수 있는 in-scope 문항을 "확인할 수 없습니다"로 거절하는 경우가 있다.
- **민원 서류 섹션 누락** — 민원 답변의 절반가량에 '필요 서류' 섹션이 없다. `civil_petition.py`가
  서류를 못 붙이는 조건을 확인할 것.
