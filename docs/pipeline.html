KDIC RAG · 데이터 파이프라인

수집에서 챗봇 입력까지,
무엇이 언제 다시 만들어지는가

58개 공개 웹페이지를 모아 근거를 인용할 수 있는 코퍼스로 만들고, 검색·평가를 거쳐 챗봇이 쓸 임베딩·청크까지 뽑는 5단계 파이프라인입니다. 핵심 규칙은 하나입니다 — 모든 단계는 다시 돌려도 같은 결과가 나오고, 본문이 실제로 바뀐 문서만 갱신으로 잡힙니다.

58페이지 (6개 업무)
198원본 HTML (본문·뒷페이지·게시글)
174평가셋 문항
2네트워크를 타는 단계

파이프라인

위에서 아래로 흐릅니다. 앞 단계의 산출물이 뒤 단계의 입력입니다.

01

src/inventory.py — 수집 대상 목록오프라인

손으로 관리하는 유일한 원천입니다. 58개 페이지의 id·URL·업무·소분류·요약을 사람이 적습니다. 업무 분류(business_function)와 계층(sub_category)이 여기서 정해지고, 검색 단계의 계층 필터가 이 값을 씁니다. 페이지를 추가하려면 여기부터 고칩니다.

PAGES · 58개 항목 dyn_table 표시 · 2개 (JS 조회표)
02

크롤러 5종 + 보조 수집기 — 원본 HTML 확보네트워크

담당자별로 크롤러가 나뉘어 있습니다(crawler_yj, crawler_dy, crawl_mistaken_remittance_jh, crawl_debt_adjustment_..._jy, crawler_hw). KDIC 사이트는 같은 페이지를 두 판본으로 서빙하므로 crawler_yj는 다시 받아 보고 inventory의 expect 문자열이 든 판본만 채택합니다. 여기까지가 유일하게 사이트를 치는 구간이고, 이후 단계는 전부 이 로컬 HTML만 씁니다.

data/raw_html/ · 58 paged/ · 109 (뒷페이지) detail/ · 31 (게시글) data/dyn/ · JS 조회표 전량 data/media_summary_jh.json · 착오송금 영상·첨부 위치
03

src/parse_raw_html.py — HTML을 본문 텍스트로오프라인

규칙 기반 변환만 씁니다(LLM 없음). 본문 컨테이너만 뽑고, 표는 행으로 펴고, 헤더·푸터 같은 사이트 공통 UI는 정확일치로만 지웁니다. 뒷페이지는 새로 나온 줄만 이어 붙입니다. 이 단계의 출력이 결정론적이라는 게 파이프라인 전체의 전제입니다 — 같은 HTML을 넣으면 언제나 바이트까지 같은 텍스트가 나옵니다.

data/text/<page_id>.txt · 58
04

src/build_corpus.py — 메타 + 코퍼스 조립오프라인

inventory의 손으로 적은 정보와, HTML에서 뽑은 링크·첨부·영상·이미지와, 본문 텍스트를 한 줄로 합칩니다. 여기서 갱신 감지가 일어납니다 — 본문 텍스트의 content_sha256을 계산해 직전 메타의 해시와 비교하고, 같으면 collected_at을 그대로 물려받습니다. 실행하면 무엇이 갱신됐는지 찍어 줍니다.

data/meta/<page_id>.json · 58 data/corpus.jsonl · 58줄 (페이지 1개 = 1줄)
05

검색·평가 + 제품 산출물 — 코퍼스를 챗봇 입력으로오프라인

chunking.py가 코퍼스를 색인 단위로 자르고(build_units — 통짜 페이지 / FAQ QA쌍 / 표 행묶음), retrieval.py가 BM25·Dense(bge-m3)·Hybrid로 검색합니다. 여기서 갈래가 둘로 나뉩니다 — 연구용은 eval_retrieval.py가 Recall·MRR로 청킹 전략을 비교하고, 제품용은 embed_corpus.py가 챗봇이 실제로 로드할 임베딩·청크를 뽑아 커밋합니다. 사이트를 치지 않습니다(첫 실행 시 bge-m3 모델 ~2GB만 내려받음).

docs/retrieval_experiment_results.md · 연구·비교 결과 data/dense_cache/*.npy · 팀 공유 Dense 임베딩 data/chunks_all.jsonl · 494 청크 (메타 포함, 챗봇 런타임 입력)

여기서부터는 아직 없습니다

파이프라인은 챗봇 입력(임베딩·청크)까지입니다. 그다음 응답 생성은 설계만 있고 코드는 없습니다.

챗봇 응답 생성 계층은 비어 있습니다. 검색·평가·임베딩(5단계)까지는 구현돼 data/dense_cache/와 data/chunks_all.jsonl이 나와 있지만, 이 청크를 받아 질문에 답을 생성하는 RAG 런타임(LLM 호출·근거 인용·media_summary_jh.json으로 위치 안내)은 아직 없습니다. 실측 근거는 docs/retrieval_experiment_results.md — 어떤 청킹이 어떤 유형 질문을 올리는지 비교해 all 모드를 제품용으로 골랐습니다. 응답 생성은 이 청크를 입력으로 새로 만들면 됩니다.

갱신 감지 — 왜 원본 HTML이 아니라 본문 텍스트를 해시하는가

이 판단 하나가 파이프라인에서 가장 자주 오해받는 지점입니다.

재수집 결과 원본 HTML 기준 (예전 방식) 본문 텍스트 sha256 (지금)
세션 토큰·주석·줄바꿈만 달라짐 갱신으로 오탐 변경 없음
사이트가 다른 판본을 서빙 (문단 빈 줄 지터) 갱신으로 오탐 변경 없음
보호한도 문구가 실제로 바뀜 갱신 갱신

실측 — dp_protlmts 라이브 재크롤 (2026-07-14)

보호한도 페이지 하나를 실제로 다시 받아 봤습니다. 사이트는 판본 2종을 내놓았고, git은 원본 HTML이 통째로 바뀌었다고 봅니다. 그런데 본문은 빈 줄 하나 말고는 글자 하나 다르지 않았습니다. 

$ python3 src/crawler_yj.py dp_protlmts --force
      판본 2종 관측 → expect 로 채택 (a4d28767)

$ git diff --numstat data/raw_html/dp_protlmts.html
30685   30672   ← git 은 "전부 바뀜" 이라고 말한다

$ python3 src/parse_raw_html.py && python3 src/build_corpus.py
완료: meta 58건 → data/meta/, corpus 58줄 → data/corpus.jsonl
  갱신 0건
  신규 0건
  collected_at: 2026-07-13  (튀지 않았다)
그래서 규칙은 이렇습니다. collected_at은 "크롤을 돌린 날"이 아니라 본문이 실제로 바뀐 것을 관측한 날입니다. 재크롤을 백 번 해도 내용이 그대로면 날짜는 안 움직입니다. 평가셋 근거가 "언제 기준인가"를 이 날짜로 말하기 때문에, 노이즈로 튀면 안 됩니다.

용어

이름이 다르면 다른 것입니다. 헷갈리는 것들만 모았습니다.

page_id
페이지 식별자. dp_protlmts처럼 생겼습니다. inventory의 id가 그대로 파일명·메타·코퍼스·평가셋 정답의 기준이 됩니다.
chunk_id / page_id
청크가 가리키는 원본 페이지. chunks_all.jsonl의 각 청크는 chunk_id(색인 단위)와 그것이 속한 page_id를 함께 싣습니다 — 검색 결과를 페이지로 접고 출처를 인용하기 위해서입니다.
content_sha256
본문 텍스트의 지문. 줄 끝 공백·CRLF·빈 줄은 무시합니다. 이 값이 같으면 내용이 안 바뀐 것이고, 그러면 collected_at을 물려받습니다.
business_function
6개 업무 중 하나. 예금자보호제도 · 예금보험금 안내 · 고객 미수령금 신청 · 착오송금 반환 신청 · 은닉재산 신고 · 채무조정 안내. 검색 시 이 값으로 범위를 좁힙니다.
expect
판본 판별 문자열. 사이트가 두 판본을 서빙하는 페이지에서, 맞는 판본에만 있는 문구입니다. 크롤러가 이걸로 골라 받습니다.
dyn_table
결과표가 JS로 채워지는 조회 화면 표시. 이 표시가 붙은 페이지만 fetch_dyntable.py가 전량 수집합니다.

다시 돌리는 법

대부분의 작업에는 네트워크가 필요 없습니다.

코퍼스만 다시 만들 때 (평소)

이미 받아 둔 HTML로 텍스트와 코퍼스를 재생성합니다. 사이트를 치지 않고 몇 초면 끝납니다.

python3 src/parse_raw_html.py
python3 src/build_corpus.py     # 갱신된 문서를 찍어 준다
python3 src/validate_testset.py # 평가셋이 코퍼스와 맞는지

검색·임베딩까지 다시 만들 때

코퍼스가 바뀌면 임베딩 캐시(파일명 = 내용 해시)가 자동 무효화됩니다. 아래를 돌려 비교 결과와 챗봇 입력을 재생성하고 재커밋합니다. 첫 실행 시 bge-m3 ~2GB를 내려받습니다. 

python3 src/eval_retrieval.py   # BM25/Dense/Hybrid × 청킹 비교표
python3 src/embed_corpus.py     # dense_cache/ + chunks_all.jsonl 재생성 → 재커밋

원본을 다시 받을 때

크롤러는 이미 받은 페이지를 건너뜁니다. 다시 받으려면 --force가 필요합니다. 받은 다음에는 반드시 위의 재적재를 이어서 돌려야 코퍼스에 반영됩니다. 

python3 src/crawler_yj.py dp_protlmts --force  # 한 페이지만
python3 src/fetch_extra.py                    # 뒷페이지·게시글
python3 src/fetch_dyntable.py                 # dyn_table 페이지 전량

python3 src/parse_raw_html.py && python3 src/build_corpus.py
순서를 지켜야 하는 이유. build_corpus.py는 직전 data/meta/를 읽어 해시를 비교합니다. 크롤만 하고 재적재를 안 하면 코퍼스는 옛 내용 그대로이고, 메타를 손으로 지우고 돌리면 전부 "신규"로 잡혀 collected_at이 오늘로 밀립니다.
