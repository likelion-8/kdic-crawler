# 연구계획서: 공공기관 안내 문서 기반 한국어 RAG 챗봇의 검색 품질 개선

팀장: 이예주
팀원: 신동엽 · 장정훈 · 홍주영 · 장희원

## 1. Problem Definition

### 어떤 task인가

RAG를 위한 문서 검색(Passage Retrieval)과 근거 기반 질의응답입니다. 사용자가 구어체로 물으면
수집해 둔 공공기관 안내 문서에서 근거를 찾고 그 근거만으로 답변과 출처 URL을 만듭니다.

### 무엇을 해결하려는가

예금보험공사(KDIC)의 기존 챗봇 '예솜24'를 RAG로 다시 만드는 프로젝트입니다. 우리 팀은 데이터 파이프라인을 맡았습니다.
예금자보호제도, 예금보험금 안내, 고객 미수령금 신청, 착오송금 반환 신청, 채무조정 안내, 은닉재산 신고.
이 6개 업무의 웹페이지를 수집·정제해서 질문에 맞는 근거를 찾을 수 있는 데이터 구조를 만드는 게 목표입니다.

데이터를 직접 만져보니 어려운 지점이 둘이었습니다.

먼저 업무 간 표현이 겹칩니다. 6개 업무가 서로 다른 제도인데 "기한", "신청 방법", "구비서류"는
어디에나 나옵니다. 6개 업무 전량(59페이지·3,662청크)에 대해 BM25로 업무 구분 없이 검색해 보니
top-1의 25.4%가 질문과 다른 업무에서 온 청크였습니다.
이 수치는 수집 원본의 결함 탓이 아닙니다. 원본 사이트에 판본 혼재 서빙 이슈가 별도로
존재하지만(고객사에 별도 보고), 두 판본의 차이는 3,662청크 중 1청크의 문장 1개뿐이고
그 청크를 옛 판본으로 치환한 재실험에서 혼입률은 0건 차이였습니다
([데이터_품질_이슈.md](데이터_품질_이슈.md) 참고). 혼입은 6개 업무가 어휘를 공유해서 생기는
코퍼스 전반의 성질이며 우리가 풀어야 할 검색 설계 문제가 맞습니다.

두 번째 어려움은 묻는 사람의 역할이 답을 바꾼다는 점입니다. 착오송금은 잘못 보낸 사람과 잘못 받은 사람이
둘 다 챗봇에 물어보는데 정답이 정반대입니다(전자는 반환지원 신청, 후자는 반환 의무와 지급명령).
비슷한 역할 축을 6개 업무에서 41개 찾았습니다. 주채무자(감면율 70~90%) vs 보증채무자(40~60%),
파산재단(1588-0037) vs KR&C(1899-0057), 증권사 CMA(비보호) vs 종금사 CMA(보호) 등입니다.
직접 재보니 "모르는 돈이 잘못 들어왔어요"(수취인) 질의가 착오송금인용 신청서·구비서류를 물어옵니다.
챗봇이 수취인에게 "반환지원 신청하세요"라고 답하게 됩니다. 정반대 안내입니다.

### 왜 중요한가

틀린 답이 바로 금전 피해로 이어집니다. 보호한도, 신청 기한, 구비서류를 잘못 안내하면 이용자가
돈을 못 받거나 기한을 놓칩니다. 그래서 일반 챗봇보다 출처 인용과 환각 방지가 훨씬 중요합니다.

실제로 수집 중에 이런 걸 봤습니다. 페이지마다 보호한도가 5천만원과 1억원으로 갈렸습니다. 낡은 페이지인 줄 알았는데
법령을 보니 둘 다 맞았습니다. 경과규정상 2025.9.1 이후 보험사고는 1억원, 그 이전 사고는 5천만원입니다.
검색이 어느 페이지를 근거로 잡느냐, 답변이 사고 시점을 붙이느냐에 따라 답의 사실성이 갈립니다.
그래서 이 프로젝트에서는 검색 품질이 답변의 사실성을 좌우합니다.

## 2. Background & Baseline

### 기존 접근

- BM25 (Robertson & Zaragoza, 2009): 어휘 일치 기반. 지금도 베이스라인으로 씁니다.
- DPR (Karpukhin et al., 2020): 질문과 문서를 같은 임베딩 공간에서 의미 유사도로 검색.
  한국어 지원 임베딩으로 multilingual-E5(Wang et al., 2024), BGE-M3(Chen et al., 2024), KoSimCSE가 있습니다.
- RAG (Lewis et al., 2020): 검색 결과를 컨텍스트로 붙여 답을 생성.
- Hybrid: BM25와 dense를 Reciprocal Rank Fusion으로 합칩니다(Cormack et al., 2009).
  BM25는 고유명사·수치에 강하고 dense는 말바꿈에 강해 서로 보완됩니다.
- HyDE (Gao et al., 2022): 짧은 구어체 질문을 가상 답변 문서로 늘려 매칭률을 높입니다.
- Lost in the Middle (Liu et al., 2023): 긴 컨텍스트의 중간을 LLM이 놓칩니다.
  그래서 검색은 작은 child 청크로, 생성은 parent 문서를 함께 주는 Parent-Child 방식을 씁니다.
- 평가는 RAGAS(Es et al., 2023), LLM-as-a-judge(Zheng et al., 2023)를 참고합니다.

### Baseline

BM25 + 글자수 청킹(500자, overlap 100) + 업무 구분 없는 평면 검색.
6개 업무 전량(59페이지·3,662청크·평가셋 378건)에서 이미 구현해 수치를 뽑아 놨습니다(`scripts/rag_probe.py`).

| Baseline (BM25 평면) | Recall@1 | Recall@3 | Recall@5 |
|---|---|---|---|
| 정답 페이지 회수율 | 43.1% | 63.5% | 70.4% |

타 업무 청크 혼입(top-1 기준) 25.4%.

## 3. Proposed Method

### 아이디어

네 가지를 순서대로 쌓습니다.

1. 계층 메타데이터 필터링. 수집할 때 모든 청크에 `business_function`(6개 업무)·`sub_category`를
   태깅해 두고 질의를 업무로 분류(라우팅)해 그 업무 안에서만 검색합니다.
2. 역할 라우팅. 질의에서 역할이 특정되지 않으면 되묻습니다. 청크의 `## 헤딩` 프리픽스에
   신청인 유형("01 본인이 찾을 경우", "02 대리인이 찾을 경우")이 보존돼 있어 이를 활용합니다.
3. Hybrid 검색. BM25에 dense 임베딩을 RRF로 결합합니다. 임베딩은 BGE-M3와 multilingual-E5를 비교해 정합니다.
4. Parent-Child retrieval. 검색은 500자 child 청크로, 생성 시 `parent_doc_id`로 원본 페이지 전문을 함께 줍니다.

### Baseline 대비 무엇이 다른가

검색 공간을 전체에서 해당 업무로 좁히고 어휘 일치만 쓰던 매칭을 어휘+의미 hybrid로 바꿉니다.
질의 쪽에 역할 축을 추가하고 생성 컨텍스트를 청크가 아니라 child + parent로 줍니다.

강조하고 싶은 건 이게 모델을 갈아끼우는 개선이 아니라 데이터 구조를 활용하는 개선이라는 점입니다.
계층 필터는 크롤링 단계에서 메뉴·브레드크럼 구조를 보존해 태깅해 놔야만 쓸 수 있고 역할 라우팅은
청킹할 때 헤딩을 조각마다 다시 붙여 놔야만 쓸 수 있습니다.
데이터 파이프라인을 어떻게 설계하느냐가 검색 품질을 좌우합니다. 이 프로젝트에서 그걸 보이려고 합니다.

### Dataset & Preprocessing

**데이터셋.** 예금보험공사 공개 웹사이트 2곳(www.kdic.or.kr, fins.kdic.or.kr)에서 직접 크롤링합니다.
사전조사에서 후보 49건을 정리했지만 그건 '사람이 손으로 적은 목록'이었습니다. 사이트 메뉴를 훑는
커버리지 검사기(`coverage_check.py`)를 만들어 대조하니 업무 구역 안에 미수집 23페이지가 더 있었습니다
(지급대행점 조회, 착오송금 지원현황, 보호대상 검색 등). 전부 추가해 59페이지로 확정했습니다.
현재 확보량은 59페이지 · 3,662청크 · 첨부 50건 · FAQ 58건 · 동적 조회표 2,797행 · 영상 자막 1건입니다.
평가셋은 원문에서 질의-답변-근거-출처URL 세트로 만들었습니다(378건, 근거검증 100%).

**데이터 특성.** 페이지당 수백~수천 자의 짧은 안내문입니다. 구비서류 표, HWP/PDF 첨부 서식,
순서도·안내그림이 섞여 있습니다. 페이지 유형도 정적 안내문 외에 FAQ 게시판, 로그인 화면, 동적 검색 화면이 있습니다.
원본 자체의 품질 문제도 있습니다. 공시이율 표는 매달 바뀌고 일부 안내문은 옛 보호한도를
시점 단서 없이 말합니다(경과규정상 과거 사고에는 여전히 맞는 수치라 삭제가 아니라 시점 표기로 다룹니다).

**전처리.** 원문 보존을 위해 전 과정을 규칙 기반으로 하고 HTML→텍스트 변환에 LLM은 쓰지 않습니다.

- HTML→텍스트: 본문 컨테이너만 추출, 표는 마크다운으로(colspan 보존). 페이지 크롬은 정확일치로만 제거.
  JS가 채우는 빈 결과표는 버립니다(안 버리면 챗봇이 "포상금 0원"이라 답합니다).
- 첨부: JS 다운로드 함수를 역해석해 받고 kordoc으로 HWP 표까지 텍스트화.
- 청킹: 500자/overlap 100. 단, 절(`## 헤딩`) 경계를 우선하고 조각마다 헤딩을 다시 붙입니다
  (안 그러면 대리인 질문에 상속인 서류를 답합니다). FAQ·조회표 행은 자르지 않습니다.
- 메타데이터: `business_function`, `sub_category`, `parent_doc_id`, `content_type`,
  페이지 요약, `collected_at`, `content_sha1`(갱신 감지).
- 비원문 자산 격리: 영상 STT는 `source_type="video_transcript"`로 따로 두고 평가셋 근거로 쓰지 않습니다.
  옛 보호한도를 시점 없이 말하는 청크에는 `time_caveat` 메타를 답니다.
- 검증: 수집본을 라이브 원문과 독립 대조하고(무오염 0건), 평가셋 근거가 원문에 실재하는지 전수 검사합니다(378/378).

### 왜 효과가 있을 것으로 예상하는가

**필터링.** 오답 원인이 "타 업무 청크 혼입 25.4%"로 이미 특정돼 있으니 검색 공간을 좁히는 건
원인을 직접 없애는 방법입니다. 실제로 필터만 켜도 **Recall@3가 63.5% → 75.1%**로 올랐습니다(+11.6%p).

**역할 라우팅.** 역할만 뒤집은 질의가 상대 역할의 문서를 물어온다는 건 이게 검색 알고리즘 문제가 아니라
질의에 역할 정보가 없어서 생긴 문제라는 뜻입니다. 검색을 아무리 개선해도 "잘못 보냈어요"와
"잘못 받았어요"가 같은 질의 벡터면 같은 결과가 나옵니다. 라우팅에서 풀어야 합니다.

**Dense/Hybrid.** BM25가 놓치는 건 "예금보험금이라는 게 정확히 뭔가요?" 같은 정의형 질문입니다.
문서 표현을 그대로 쓰지 않고 말을 바꿔 묻는 유형이라 의미 매칭이 보완해 줄 것으로 봅니다.

**Parent-Child.** 조건·예외가 top-k 밖으로 밀려 답이 틀리는 경우를 막습니다.
근거 무결성이 378/378이라 근거는 이미 단일 청크 안에 온전히 들어 있습니다. 즉 검색만 맞히면 답할 수 있습니다.
parent를 함께 주는 건 조건 누락 방지용입니다.

## 4. Experiment Design

### 어떤 비교를 할 것인가

| 실험 | 구성 | 무엇을 확인하나 (판정 기준) |
|---|---|---|
| E1 | BM25 평면 (baseline) | 시작점. 사전 실측 Recall@3 63.5% 재현 |
| E2 | E1 + 계층 필터 | 필터 효과. 사전 실측 75.1% (+11.6%p) 재현 |
| E3 | Dense 단독 + 필터 | 임베딩 2종(multilingual-E5·BGE-M3) 비교. BM25 하한선(75.1%)에 근접하는가 |
| E4 | Hybrid(RRF) + 필터 | **E2·E3 둘 다 넘는가.** 못 넘으면 hybrid 채택 안 함 |
| E5 | E4 + 역할 라우팅 | 역할 축 검사 `[RB]` **3/4 → 4/4** 회복되는가 |
| E6 | E5 + Parent-Child + 답변 생성 | E2E 지표 4종(아래) — 특히 근거 없는 단정 답변 0건 |

E1·E2는 이미 잰 수치의 재현 확인이고 본 실험은 E3부터입니다. 검색 지표는 정답 페이지 회수율과
근거 청크 회수율(사전 실측 @3 62.4%로 페이지 회수율보다 12.7%p 낮습니다)을 반드시 같이 봅니다.

추가로 두 가지를 봅니다.

- 범위 밖 질의: 코퍼스에 답이 없는 질문에 지어내지 않고 거절하는가.
  참고로 BM25 점수 임계값으로 범위 밖을 거르는 건 안 된다는 걸 확인했습니다. 범위 안/밖 점수 분포가 겹칩니다.
- 역할 혼동 질의: 같은 사실을 묻되 역할만 바꾼 쌍(착오송금인/수취인 등)을 넣어
  서로 다른 답이 나오는지 봅니다. 역할 축 41개에서 대표 쌍을 뽑습니다.

### Evaluation Metrics

**검색 단계**: Recall@1/3/5, MRR. 정답 기준은 평가셋의 `page_id`입니다.

**E2E 응답 단계**: 네 가지를 봅니다.

- 정답성: LLM-as-a-judge가 생성 답변을 정답·원문 근거와 대조해 correct/partial/wrong 판정
- 출처 정확도: 인용 URL이 정답 페이지 URL과 일치하는가
- 환각률: 컨텍스트에 없는 사실을 답변에 넣은 비율
- 거절 품질: 범위 밖 정상 거절률 / 범위 안 과잉 거절률

지표 자체를 어떻게 믿을 것인가.
평가셋의 근거는 사람이나 모델의 주장이 아니라 스크립트가 원문과 문자열 대조로 전수 검증합니다
(`testset_build.py`, 378/378 통과). 근거가 원문에 없으면 그 문항은 버립니다.
LLM judge는 표본을 팀원 2인이 수작업 채점해 일치율을 확인하고 어긋나는 유형이 보이면 프롬프트를 고칩니다.
그리고 검색 지표와 E2E 지표를 반드시 같이 봅니다. 같은 정의가 여러 페이지에 있어 "다른" 페이지가
잡혀도 답은 맞는 경우가 있어서 페이지 회수율만 보면 응답 가능성을 과소평가합니다.
